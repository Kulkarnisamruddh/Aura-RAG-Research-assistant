# Aura: Technical Project Report & Interview Guide

This document provides a deep dive into the architecture, design decisions, and engineering trade-offs made during the development of **Aura**.

**Developed By:** Samruddhi Kulkarni (Final year AI&DS student, MIT Chhatrapati Sambhajinagar)

---

## 1. System Overview
Aura is a full-stack **Retrieval-Augmented Generation (RAG)** application. It allows researchers to upload documents (PDF, TXT, MD), convert them into high-dimensional vector embeddings, and perform context-aware chat with ultra-low latency.

### Core Stack
- **Frontend:** React 19, Vite, Vanilla CSS (Glassmorphism design).
- **Backend:** Node.js (Express) with Server-Sent Events (SSE) for streaming.
- **Database:** Supabase (PostgreSQL) + **pgvector** extension.
- **AI/ML:** 
  - **Embeddings:** `all-MiniLM-L6-v2` (Local execution via Xenova/Transformers.js).
  - **LLM:** Llama-3.3-70B (Groq) for lightning-fast inference.

## 2. Technical Deep Dive

### A. The Embedding Pipeline (The "A" in RAG)
Instead of using expensive APIs like OpenAI for embeddings, Aura generates them **on the backend** using a local quantized model.
- **Model:** `Xenova/all-MiniLM-L6-v2`.
- **Dimensions:** 384.
- **Optimization (Edge Computing):** To support deployment on limited-resource servers (like Render Free Tier), Aura uses **Browser-Side Embeddings**. The AI model is downloaded and executed in the user's browser via Web Workers. This ensures the backend remains lightweight, avoids memory crashes, and scales infinitely with the number of users.

### B. Vector Search & Storage
Document chunks are stored in a PostgreSQL table with a `vector` column type.
- **Search Method:** Cosine Similarity (`<=>` operator).
- **Optimization:** The system uses an RPC function `match_document_chunks` to perform search directly in the database layer, minimizing data transfer between the DB and the server.

### C. Streaming Infrastructure
To solve the UX "latency gap," Aura implements **Server-Sent Events (SSE)**.
- **Mechanism:** The backend keeps an HTTP connection open and "pushes" tokens as they are generated by Groq.
- **Benefit:** Reduces "Time to First Token" (TTFT) from seconds to milliseconds.

## 3. Security & Scalability

### Security: Document Isolation
Aura implements **Row-Level Security (RLS)**.
- **Mechanism:** Every request from the frontend passes a Supabase JWT. The backend uses this JWT to instantiate an authenticated Supabase client.
- **Result:** Even if an attacker knows a document ID, they cannot access it unless they are the owner. The database itself enforces this rule.

### Scalability: Stateless Design
The backend is stateless. It does not store sessions in memory; it relies on JWTs.
- **Future Growth:** This allows the Node.js server to be scaled horizontally (multiple instances behind a load balancer) without any "sticky session" issues.

## 4. Potential Interview Questions & Answers

### Q: Why use fixed-size chunks instead of whole documents?
**A:** "LLMs have a context window limit (e.g., 128k tokens). By chunking text into 300-word segments, we can retrieve only the *most relevant* pieces of information, saving tokens and improving the focus of the answer."

### Q: How do you handle 'Hallucinations'?
**A:** "Aura uses strict **System Prompting**. We instruct the model to only use the provided context. If the answer isn't in the context, the model is told to say 'I cannot find the answer in the documents' rather than making something up."

### Q: Why Groq over OpenAI?
**A:** "Speed. Groq's LPU (Language Processing Unit) architecture allows for inference speeds of 200+ tokens per second, which is critical for a smooth research experience."
